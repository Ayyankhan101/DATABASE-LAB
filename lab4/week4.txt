Three problems appear immediately.
Redundancy. Ali's name, email, and city are repeated on every row of every order he ever places. A
customer who places 20 orders has their details stored 20 times.
Update anomalies. If Ali moves from Peshawar to Islamabad, you have to update every single row
that mentions him. Miss one and your data is inconsistent.
Deletion anomalies. If you delete the only order containing a certain product, you lose all
information about that product.
The solution is to split the data into separate tables — one for customers, one for products, one for
orders — and connect them using keys. This is called normalisation, and you have been living with
the results of it since Week 2 without knowing it.

Foreign Keys
A foreign key is a column in one table that refers to the primary key of another table. It is the
mechanism that connects tables together.
In your schema:
-- The orders table has a foreign key pointing to customers
orders.customer_id → customers.customer_id
-- The order_items table has foreign keys pointing to two tables
order_items.order_id
→ orders.order_id
order_items.product_id → products.product_id

Foreign keys enforce referential integrity — they prevent you from creating an order for a customer
that does not exist, or adding an order item for a product that has been deleted. The database itself
protects your data from becoming inconsistent.

2.1 INNER JOIN
An INNER JOIN returns only the rows where the join condition matches in both tables. If a row in
the left table has no match in the right table, it does not appear in the result. Same for the right table.
-- The most common join: connect orders to the customers who placed them
SELECT o.order_id,
c.name
AS customer_name,
c.city,
o.order_date,
o.total_amount,
o.status
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id
ORDER BY o.order_date DESC;

Notice the aliases: orders o and customers c. When joining tables you almost always use
short aliases because you need to prefix column names to avoid ambiguity — both tables might
have a column called id, and PostgreSQL would not know which one you mean without the prefix.
The ON clause is where you specify the relationship. It is saying: "Match each row in orders to the
row in customers where the customer_id values are equal."
-- Three-table join: bring in product details alongside order items
SELECT o.order_id,
c.name
AS customer_name,
p.product_name,
p.category,
oi.quantity,
oi.unit_price,
oi.quantity * oi.unit_price AS line_total
FROM order_items oi
INNER JOIN orders
o ON oi.order_id
= o.order_id
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN products p ON oi.product_id = p.product_id
ORDER BY o.order_id, p.category;

This is the full picture of a transaction: who bought it, what they bought, how many, and at what
price. This is the kind of query that feeds a sales report or a recommendation model's training data.

2.2 LEFT JOIN
A LEFT JOIN returns all rows from the left table, plus any matching rows from the right table.
Where there is no match, the right-side columns come back as NULL.
-- Find ALL customers, including those who have never placed an order
SELECT c.customer_id,
c.name,
c.city,
c.signup_date,
COUNT(o.order_id)
AS total_orders,
COALESCE(SUM(o.total_amount), 0) AS total_spent
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.name, c.city, c.signup_date
ORDER BY total_spent DESC;

If you used an INNER JOIN here, customers who have never ordered would simply disappear from
the results. With LEFT JOIN they appear with a 0 for total_spent and 0 for total_orders. In data
science this distinction matters enormously — a customer who signed up but never bought anything
is a completely different data point from one who never existed.

2.3 RIGHT JOIN
A RIGHT JOIN is the mirror of LEFT JOIN — it returns all rows from the right table and matching
rows from the left. In practice, most developers rewrite RIGHT JOINs as LEFT JOINs by swapping
the table order, because it is easier to reason about. Both produce identical results.
-- Find ALL products, including those that have never been ordered
-- Using RIGHT JOIN
SELECT p.product_name,
p.category,
p.price,
COUNT(oi.item_id) AS times_ordered
FROM order_items oi
RIGHT JOIN products p ON oi.product_id = p.product_id
GROUP BY p.product_id, p.product_name, p.category, p.price
ORDER BY times_ordered DESC;
-- Exact same result, written as LEFT JOIN (preferred)
SELECT p.product_name,
p.category,
p.price,
COUNT(oi.item_id) AS times_ordered
FROM products p
LEFT JOIN order_items oi ON p.product_id = oi.product_id
GROUP BY p.product_id, p.product_name, p.category, p.price
ORDER BY times_ordered DESC;

Products that have never been ordered will appear with a count of 0. This is a classic inventory
question: what are we stocking that nobody buys?

2.4 FULL OUTER JOIN
A FULL OUTER JOIN returns all rows from both tables. Where there is a match, the columns from
both sides are populated. Where there is no match on either side, the unmatched side comes back as
NULL.
-- All customers AND all sessions, even if one side has no match
SELECT c.name,
c.city,
s.session_date,
s.device,
s.converted
FROM customers c
FULL OUTER JOIN user_sessions s ON c.customer_id = s.customer_id
ORDER BY c.name, s.session_date;

FULL OUTER JOIN is less common than INNER or LEFT, but it is useful when you want to audit
two datasets against each other — finding rows that exist in one source but not the other, which is a
very common data quality task.

2.6 A Word on Normalisation
Your e-commerce schema is in Third Normal Form (3NF). You do not need to memorize the formal
definitions right now, but you should understand what each level means in plain terms:
First Normal Form (1NF): Every column contains a single atomic value. No repeating groups. A
column called "products_bought" that stores "Mouse, Keyboard, Book" in one cell violates 1NF.
Your schema satisfies this — order_items exists precisely to avoid storing multiple products in one
orders column.
Second Normal Form (2NF): Every non-key column depends on the entire primary key, not just
part of it. This matters when you have composite primary keys. Your schema satisfies this.
Third Normal Form (3NF): Every non-key column depends only on the primary key, not on
another non-key column. A classic violation: storing both city and postal_code in customers where
postal_code determines city. If you ever store something that can be derived from a non-key
column, you are likely violating 3NF.
Why does this matter for data science? Because poorly normalised data creates the data quality
problems you will spend hours cleaning before you can build a model. Understanding normalisation
helps you design schemas where data quality problems cannot happen in the first place.

Query 3 — Customers who have never ordered
Use a LEFT JOIN to find customers who signed up but have never placed any order. This is the first
step in any churn or re-engagement analysis.
SELECT c.customer_id,
c.name,
c.email,
c.city,
c.signup_date
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL
ORDER BY c.signup_date;

The trick here is WHERE o.order_id IS NULL. After a LEFT JOIN, rows from the left side
that had no match come back with NULLs on the right side. Filtering for those NULLs is how you
isolate the non-matching rows. This pattern — LEFT JOIN followed by WHERE right_side IS
NULL — is one of the most useful patterns in SQL.

SELECT c.name,
c.city,
COUNT(o.order_id)

For each city, show how many customers are based there, how many orders those customers have

SELECT c.city,
COUNT(DISTINCT c.customer_id)
AS num_customers,
COUNT(DISTINCT o.order_id)

SUM(o.total_amount)
AS total_revenue,

Query 9 — Session and purchase behaviour together
Revisit the session-to-purchase analysis from Week 3, but this time use a proper join structure
instead of separate CTEs. For every customer who has at least one session, show their name, city,
number of sessions, whether they have ever purchased, and if so their total spend.
SELECT c.name,
c.city,
COUNT(DISTINCT s.session_id)
AS total_sessions,
COUNT(DISTINCT o.order_id)
AS total_orders,
COALESCE(SUM(DISTINCT o.total_amount), 0) AS total_spent,
CASE
WHEN COUNT(DISTINCT o.order_id) > 0 THEN 'Converted'
ELSE 'Not Converted'
END
AS customer_status
FROM customers c
INNER JOIN user_sessions s ON c.customer_id = s.customer_id
LEFT JOIN orders
o ON c.customer_id = o.customer_id
AND o.status = 'delivered'
GROUP BY c.customer_id, c.name, c.city
ORDER BY total_sessions DESC;

Notice the join condition AND o.status = 'delivered' is written inside the ON clause
rather than in WHERE. This is intentional — putting it in WHERE would turn the LEFT JOIN into
an effective INNER JOIN. When you want to filter one side of a LEFT JOIN without eliminating
non-matching rows, the condition goes in ON.

Query 10 — FULL OUTER JOIN: data completeness audit
Write a query using FULL OUTER JOIN to identify any mismatches between the customers table
and the user_sessions table. Find customers who have no session data, and any orphaned session
records with no matching customer.
SELECT c.customer_id
AS customer_table_id,
c.name,
s.customer_id
AS session_table_cust_id,
COUNT(s.session_id)
AS session_count,
CASE
WHEN c.customer_id IS NULL THEN 'Orphaned session — no customer
record'
WHEN s.customer_id IS NULL THEN 'Customer with no sessions'
ELSE 'Matched'
END
AS audit_status
FROM customers c
FULL OUTER JOIN user_sessions s ON c.customer_id = s.customer_id
GROUP BY c.customer_id, c.name, s.customer_id
ORDER BY audit_status, c.name;

In real data work this kind of audit query runs before any analysis. If you find orphaned records,
they indicate a data pipeline problem. If you find customers with no sessions, those might be
legitimate or might indicate a tracking gap. Either way, you need to know before you build a model.

Part 4 — Performance Awareness
JOINs are where query performance gets genuinely interesting.
EXPLAIN ANALYZE
SELECT c.name, o.order_id, o.total_amount
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id
WHERE o.status = 'delivered';

Look for Hash Join or Nested Loop in the output. These are the two main strategies PostgreSQL
uses to execute joins.
A Nested Loop join works like two nested for-loops: for each row in table A, scan table B for a
match. On small tables this is fine. On large tables it becomes O(n²) — extremely slow.
A Hash Join builds a hash table from one side, then probes it for each row from the other side.
Much faster on large datasets, but requires memory to build the hash table.
On your small dataset PostgreSQL will probably choose Nested Loop because it is faster for small
data. On a table with millions of rows it would switch to Hash Join automatically. EXPLAIN
ANALYZE lets you see what it chose.
For your deliverable, run EXPLAIN ANALYZE on Query 2 (the four-table join). Note every
operation type that appears and write one sentence explaining what each one does.

Quick Reference
-- ── JOIN skeleton ─────────────────────────────────────────
SELECT a.col1, b.col2
FROM
table_a a
-- left table
JOIN
table_b b
-- right table (INNER by default)
ON a.id = b.a_id
-- join condition
WHERE ...
-- row filter (applied after join)
ORDER BY ...;
-- ── JOIN types ────────────────────────────────────────────
INNER JOIN -- only matching rows from both sides
LEFT JOIN
-- all rows from left, NULLs where no right match
RIGHT JOIN -- all rows from right, NULLs where no left match
FULL OUTER JOIN -- all rows from both sides, NULLs where no match
-- ── Find rows with no match (anti-join) ───────────────────
SELECT a.*
FROM
table_a a
LEFT JOIN table_b b ON a.id = b.a_id
WHERE b.id IS NULL;
-- rows in A with no match in B

o

JOIN

c
ON o.customer_id = c.customer_id
JOIN

p
ON oi.product_id = p.product_id;

COUNT(*)
-- every row, including duplicates
COUNT(o.order_id)
-- non-NULL values only
COUNT(DISTINCT o.order_id) -- unique values only

WHERE o.status = 'delivered'
